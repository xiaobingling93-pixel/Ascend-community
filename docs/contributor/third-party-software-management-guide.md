# Ascend 开源与第三方软件管理规范

---

## 目的

Ascend社区将满足 [Open Source Definition](https://opensource.org/docs/osd) 定义的软件认同为开源软件。为提供易用、高质量的开源软件，确保社区项目的整体质量，特拟定本规范，供Ascend社区贡献者参考。

---

## 范围

本规范适用于所有引入的开源与第三方软件。

---

## 软件引入原则

为实现对软件生命周期的维护和管理，在引入时请参考如下原则：

1. 软件应该有明确的引入理由，尽量重用，不引入同一软件的多个版本，原则上引入源码而非二进制

2. 引入软件从开源软件官网或官网指定的代码托管地址获取；禁止引入无许可证、许可证要求无法履行、有知识产权问题的软件包，高风险license与存在已知未修复高危漏洞的开源软件谨慎引入

3. 应当完整保留引入软件的官方代码仓目录结构、许可证及Copyright信息，不要修改开源与第三方软件的原始许可证与Copyright信息

4. 慎选来自个人、小型社区或组织、两年内未发布版本、无明确版本计划、社区提交了有效的Bug或PR，但半年以上未响应、官网明确指出不再维护等的开源软件

5. 软件引入后有明确的维护责任团队

## 软件引入流程

### 软件引入前检查

| 检查点 | 说明 |
|--------|------|
| 归一化 | 1、原则上一款软件只引入一次。 |
| 来源可靠 | 1、开源软件都应该从开源软件官网获取或官网指定的代码托管地址获取。 |
| 规范化软件名称 | 1、 软件名称必须和官网/社区保持一致，不可随意命名。<br>2、 不允许以软件包中的子模块作为软件名。<br>3、 当软件是某个语言的开发库时，我们允许追加python-、perl-等前缀予以规范化管理。 |
| 社区运营状态检查 | 1、软件来自知名社区或组织，社区或组织通过发布公告、修改软件仓库状态、将仓库放到特定目录下等方式告知停止维护的，不建议引入。<br>2、软件来自个人、小型社区或组织，两年内未发布版本（含正式版本与测试版本），无明确版本计划，社区提交了有效的Bug、PR、CVE，但是半年以上未响应的，不建议引入。<br>3、社区运营状态不明确，通过Issue 或者邮件等方式询问社区是否继续维护，半年以上未响应或者答复停止维护的，不建议引入。 |
| 官网必填 | 1、软件官方网址填写规范，使用软件供应商提供的网址；无单独正式官网的情况下，提供主流代码托管平台上对应的项目网址，如github。<br>2、不能使用maven、mvnrepository、springsource等托管库作为官方网址。 |
| 软件包信息提供 | 1、必须提供官方提供的源代码包的下载地址，以达到可溯源。<br>2、如果有需要二进制包，需要提供官方的二进制包下载地址。 |
| license检查 | 1、待引入软件是否有license。<br>2、入库时填写的license是否和官网保持一致；是否和软件包中license保持一致；高风险license的开源软件谨慎引入。 |
| copyright检查 | 1、通过官网、社区、代码托管网站、源码包、发布件中等地方获取并提供copyright信息。 |

### 提交申请

如需要引入新的软件，请在申请材料中包含如下内容：

#### 1. 自检表

| 检查项 | 填写指导 | 自检结果示例 |
|--------|----------|--------------|
| 软件名称 | 描述该软件官方名称以及引入后的仓名 | softwarename |
| 软件官网地址 | 描述该软件官方网站链接地址 | <https://softwaresite> |
| 软件版本号 | 描述该软件要引入的版本号，版本号为其社区正式发布的版本号，不要随意修改；未正式发布的版本不建议引入 | 1.0.0 |
| 软件版本发布日期 | 描述该软件要引入的版本的社区发布日期 | 2021.01.01 |
| 软件版本地址 | 描述该版本的官方下载链接地址与GitCode镜像仓库地址（便于高速下载），注意该地址必须为能定位到该具体版本发布包的下载URL | <https://gitcode.com/softwarecodesite/v1.0.0.zip> |
| 软件许可证 | 描述该版本的官方许可证名称及许可证文件的相对路径，如果是多许可证要完整描述，并说明各许可证的关系，如是And还是Or，或者是不同目录对应不同许可证 | Apache-2.0 |
| 软件生命周期 | 描述该软件是否有LTS版本，多长时间发布一个版本，最近一年社区代码提交及Issue解决情况，是否有告知停止维护或演进 | 无LTS版本，6个月发布一个版本，近6个月有10次代码提交 |
| 软件安全漏洞 | 描述该软件存在的业界公开的安全漏洞列表，包括漏洞号、级别、漏洞链接地址、是否已有补丁或解决方案 | 无业界公开漏洞 |
| 业务场景 | 描述该软件被哪些仓使用，解决什么业务场景的问题 | 被XX仓静态链接使用，提升YYY能力 |
| 归一化 | 描述社区是否已有此软件或类似软件，业界类似软件有哪些，为什么要新引入该软件或该版本 | 本社区还未引入此软件，业界相似软件有B、C，只有本软件许可证友好，且该软件生态较好，X、Y等公司也在使用 |
| 许可证兼容性 | 描述使用该软件有哪些进程，各进程的许可证是什么，与要引入软件的许可证是否兼容。 | 此软件在用户态X进程中，静态链接使用，该进程的许可证为Apache-2.0，与该软件许可证一致，无兼容性问题 |
| 责任人 | 描述该软件引入到本社区后的SIG名称及维护责任人的GitCode用名及邮箱 | SIG XXX，Zhangsan，<Zhangsan@xyz.com> |

### 新增开源与第三方软件评审

在引入该软件的目标仓库对应的SIG的TC（Technical Committee，技术委员会）会议上进行评审。

### 集成方式选择

评审通过后，根据定制修改需求选择集成方式：

1. **大量定制修改场景**：当需要对软件进行大量定制修改时，在引入该软件的目标仓库对应的SIG组下建仓，按照[开源与第三方软件建仓及分支命名指导](third-party-repo-branch-guide.md)进行独立建仓，后续维护由该SIG组负责。
2. **少量修改或无需修改场景**：
   - 如果软件通过Git管理：使用Git Submodule方式集成，优先使用GitCode国内镜像链接，通过patch方式在仓库内进行少量定制修改或漏洞修复，文件结构如下：

     ```text
     仓库根目录/
     ├── .gitmodules          # Git Submodule配置文件，放置在根目录下
     └── third_party/
         ├── 第三方开源软件目录/
         └── patch/
             └── 第三方开源软件名/
                 └── xxx.patch
     ```

   - 如果软件通过包管理器管理：添加到对应的包管理清单中，如requirements.txt、pom.xml等。

## 软件版本管理

软件版本变更可以按照使用方的实际需求进行，如漏洞修复、新功能引入等。但在进行版本变更前，需要判断上下游软件对该软件的使用情况，评估版本变更对依赖项目的影响，确保不会出现兼容性问题或影响其他项目的稳定性。

## 软件退出原则

当满足以下两个条件时，软件退出申请可以被立即执行，对应文件将从项目中直接删除。

1. 软件的License变化，或者其他法律法规影响了目前正在使用的版本，导致本社区因为法务风险，不能继续集成该软件。

2. 存在恶意代码或严重安全隐患且本社区无能力修复的，要求软件被立即移除。

除以上描述两种场景外，其它场景对软件的退出实行过程化管理：

1. 随着技术演进与发展，软件因技术陈旧或架构落后，不能满足现有的应用场景被其他更优秀的软件所取代。

2. 已经集成的版本过于老旧，且软件新版本License或其他法律法规限制导致本社区软件不能升级新版本。

3. 软件来自知名社区或组织，社区或组织通过发布公告、修改软件仓库状态、将仓库放到特定目录下等方式告知停止维护的。

4. 软件来自个人、小型社区或组织，两年内未发布版本（含LTS版本、Release版本、Beta版本等），无明确版本计划，社区提交了有效的Bug或PR，社区半年以上未响应的。

5. 社区运营状态不明确，通过Issue或者邮件等方式询问社区是否继续维护，社区半年以上未响应或者答复停止维护的。

如果软件符合以上任何一条退出条件，开源SIG与相应SIG首先分析该软件在当前社区中被依赖、被使用的情况。

1. 如果存在依赖关系，且短时间内不能解除，我们建议SIG新建分支代码仓，并主动进行社区维护动作。

2. 如果不存在依赖关系，或者短时间内可以解除，则责任SIG将软件从社区正式发行中移出，并在相应的Release Notes中说明移除的原因及影响。
