# 社区 Issue 处理流程指导

为规范社区及项目中 Issue 的管理流程，明确 Issue 从创建、处理到闭环全生命周期的操作标准，统一标签使用规则、响应时效要求及沟通规范，提升 Issue 处理效率与质量，保障社区贡献者与维护者的协作顺畅，特制定本规范。本文档适用于所有社区成员及项目维护者在 Issue 相关操作中的行为指引。

# 1. 核心标签定义与说明

|标签类别|标签名称|含义说明|
|---|---|---|
|处理状态|triage review|待分类审核，需维护者初步评估 issue 性质、优先级等|
||triaged|已完成分类审核，明确了类型、优先级和归属模块，进入下一处理阶段|
||assigned|已分配给具体开发者处理|
||pending|等待依赖项完成（如其他 PR 合并）或外部反馈，暂时无法推进，标记为 pending 的不会统计入 Issue 关闭时长，但需要评论 pending 合理理由|
||fixed|问题已修复，通常对应 PR 已合并|
||wontfix|明确不会修复（如因设计目标、兼容性等原因），会说明理由并关闭|
||duplicate|与已有 issue 重复，会被关联到原 issue 并关闭|
||invalid|无效 issue（如误报），会被关闭|
|问题类型|bug|功能错误或不符合预期的行为|
||feature|新功能或改进建议|
||query|对现有功能的优化（区别于全新功能）|
|优先级|high priority|高优先级，需尽快处理（如严重 bug、影响核心功能）|
||medium priority|中优先级，按常规流程处理|
||low priority|低优先级，可延后处理（如边缘场景问题）|
|模块归属|module: XXX*|标记 issue 所属模块为 XXX*（需替换为具体模块名称）|
|其他常用|help wanted|需要社区贡献者协助解决的问题|
||good first issue|适合新手贡献者的入门级问题，通常难度较低、文档清晰|

*注：在创建 Label 时，此占位词需替换为具体模块名称

# 2. 标签标记阶段规范

| 标签类别 | 首次响应应标记 | 责任人审视完成需标记 | 开发阶段 | 关闭需标记 |
|---------|---------------|-------------------|---------|-----------|
| 处理状态 | triage review | triaged, pending, duplicate, invalid | assigned | fixed, wontfix |
| 问题类型 | - | bug, feature, query | - | - |
| 优先级 | - | high priority, medium priority, low priority | - | - |
| 模块归属 | module: XXX* | - | - | - |
| 外部贡献 | - | help wanted, good first issue | - | - |

*注：在创建 Label 时，此占位词需替换为具体模块名称

# 3. 全流程标签使用指南

## 3.1 首次响应阶段：标签初设

标记 `triage review` 表示问题已有对应人员进行查看，分配归属模块标记：`module: xxx`

- 如社区中不区分模块，跳过标记 `module`

- 如初步分析无法确定归属模块，可由下一阶段审视责任人在具体分析完问题后进行标记

此阶段完成后 Issue 需要有对应的跟踪处理人。

## 3.2 审视分析阶段：标签细化

根据 Issue 具体内容分析完成后，对 Issue 进行：

- 标记为 `triaged` 表示已完成分析，以及下一步行动 `pending`/`duplicate`/`invalid`：
  - `pending` 需给出具体等待的合理原因；
  - `duplicate` 需关联已有 issue 或 PR 后进行关闭；
  - `invalid` 需给出合理解释后关闭，标记 `wontfix`；

- 标记问题的类型：`bug`/`feature`/`query`；
- 标记问题的优先级：`high priority`/`medium priority`/`low priority`，对于 `low priority` 需在评论中给出理由；
- 如期望外部开发者贡献标记 `help wanted`，适合新手的简单问题额外标记 `good first issue`。

## 3.3 开发处理阶段：标签更新

指派 Issue 对应开发者，标记 `assigned`。

## 3.4 问题闭环阶段：标签终设

标记最终处理状态：
- 已修复问题标记：`fixed`
- 不修复该问题或非问题标记：`wontfix`，并给出不修复的具体原因，需要与 Issue 提交者达成一致

# 4. 响应与关闭 SLA 要求

## 4.1 响应时效要求

| 具体场景 | 时效标准 |
|---------|---------|
| 首次响应 | SLA < 24 小时 |
| 后续交互 | SLA < 24 小时 |

## 4.2 关闭时效要求

Issue 可分为 3 类（咨询类/bug 类/需求类），平均关闭时间要求如下：

| Issue 类型 | 平均关闭时间 |
|-----------|-------------|
| 咨询类 | ≤ 2 天 |
| Bug 类 | ≤ 7 天 |
| 需求类 | ≤ 30 天 |

当用户问题明确，给出解决方案或解决方案时间，并提醒用户关单。


# 5. 超期 Issue 自动处理

社区机器人会自动处理超期 Issue，处理流程如下：

- Issue 被标记为 `pending` label 且无新增评论超 30 天，自动添加 `stale` label；
- Issue 被标记为 `stale` label 且无新增评论超 14 天，将此 Issue 关闭。

标记为 `pending` 后超期需要关闭的 Issue 统一由社区 bot 进行操作，社区 Maintainer/Committer 不需进行手动处理。

# 6. 有效回复，避免无效沟通

任何给予 Issue 提交者模糊、不确定、流程、方案不明确的回答，都会降低开发者对社区的热情，造成不必要的误解。避免提供只有评论者自己了解，但 Issue 参与者无法感知到后续方向的简短回复，例如：

- “收到”――下一步是谁处理，联系谁应该？
- “已解决”――在什么版本解决的，是否有 PR 连接或过往 Issue 连接？
- “待排版本”――具体是哪个版本，谁在排，排的流程是什么？

需提供明确的下一步指引，告诉提交者当前状态是什么，下一步他需要怎么做。
