# 社区安全设计规范

基于社区产品的实际业务应用场景，参考业界最佳安全实践，制定本社区安全设计规范要求，用于牵引、指导和约束社区的安全设计工作，提升社区产品安全质量。

## 1、个人数据保护和合规

1.1  个人数据的搜集和处理需要获得用户本人同意。获取敏感个人数据或者对用户画像时需获得用户的单独同意。应至少在产品的客户手册等客户面资料中说明搜集的个人数据的范围、用途、保护措施、存留期等，并在发生变更时及时更新上述说明。

1.2  个人数据的搜集需遵循最小化原则，禁止搜集不必要的个人数据；

1.3  个人数据应该加密存储，并提供必要的安全保护机制（如访问权限控制、访问日志记录等）；存储的个人数据超过声明的存留期后应立即删除。

1.4  禁止未经授权将个人数据传出客户网络；禁止个人数据跨境转移。

注：

Z1：个人数据：是指以电子或者其他方式记录的与已识别或可识别的自然人有关的各种信息。匿名化后的数据不属于个人数据。假名化后的数据仍属于个人数据。

Z2：敏感个人数据：是指一旦泄漏或者非法使用，容易导致自然人的人格尊严受到伤害或者人身、财产安全受到危害的个人信息，包括生物识别、宗教信仰、特定身份、医疗健康、金融账户、行踪轨迹等信息，以及未成年人的个人信息（未成年人的标准在不同国家有不同定义。在中国，未满14周岁为未成年人）

## 2、系统接入安全

2.1  所有能对系统管理的人机接口以及机机接口必须有接入认证机制，禁止存在绕过正常认证机制直接进入系统进行系统管理的隐秘管理通道。

2.2  禁止将认证凭据（如口令/密码/私钥等）包含在发布的软件包中，所有接入认证凭据都必须支持修改，建议支持加密存储这些认证凭据。

## 3、系统安全

3.1  安全方案设计要遵从操作系统的安全框架，不要违反系统现有的安全设计原则，不要破坏系统原有安全框架。

3.2  产品对外提供能够被远程访问的服务和进程，应使用系统低权限账号运行，不应使用“root”等特权账号运行。

3.3 禁止在日志、调测信息中打印口令、秘钥等敏感信息。

3.4 产品对外发布的各类软件或独立AI软件包，需具备完整性校验机制（如数字签名，或者哈希值校验等）

## 4、数据传输安全

4.1  在不同实体、节点或者网络之间传递机密数据（如个人数据、口令或秘钥、运营业务关键数据等）时，若存在泄漏风险，系统应支持使用通用加密通道传输，或者支持将机密数据加密后再传输。建议系统缺省部署和开启这些安全机制。

4.2  使用安全传输通道保护数据时，建议使用业界标准的传输安全协议（如IPSec、MACSec、HTTPS等）。

## 5、密码学安全

5.1  系统中应该使用业界公开的、标准的密码学算法，和使用经过权威机构（如ISO、RFC等）推荐或认可的密码学算法库。

5.2  禁止使用已经被破解的、或者存在严重安全漏洞的密码学算法和协议，如禁止使用被破解的DES、MD5等算法，禁止使用 RSA 1024 低秘钥强度算法，禁止使用TLS1.0等被破解的安全协议等。当这些不安全密码学算法和协议用于非安全用途时可不遵守本条要求。

5.3  密码算法中使用的随机数，必须是密码学意义上的安全随机数。例如，如下方式生产的随机数可以被认为是安全随机数：RAND byes（OpenSSL）、/dev/random文件（类Unix）、/dev/urandom文件（Linux Kernel 5.18 及其之后的版本）
