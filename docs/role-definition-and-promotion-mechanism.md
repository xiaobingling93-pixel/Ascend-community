# Ascend 社区角色定义及晋升机制

为明确 Ascend 社区各角色的核心权责与成长路径，规范成员晋升、退出流程，保障社区健康有序运营，特制定本机制。本文档适用于 Ascend 社区各 SIG（特别兴趣小组）及项目代码仓库。

## 1. 角色定义

| 角色 | 核心定义 | 核心权责 |
|------|----------|----------|
| **用户（Users）** | 使用Ascend社区项目的广大用户群体 | 反馈使用问题、提出功能需求，参与社区调研与产品体验反馈 |
| **贡献者（Contributors）** | 拥有一定编程经验，主动参与社区建设的开发者 | 提交代码/文档PR、解答Issue疑问、参与技术方案讨论、发表技术文章、策划社区活动 |
| **提交者（Committers）** | 拥有特定SIG子领域代码仓库写权限的核心贡献者 | 负责模块设计与评审、代码审核与维护，处理社区Issue，指导Contributors提升开发技能 |
| **维护者（Maintainers）** | 负责特定SIG运营与维护的核心负责人 | 定义SIG工作范围与目标，吸纳并培养Committer，推动项目孵化与推广，定期向技术指导委员会（TSC）汇报运营进展并落实改进 |

注：Committer的权限与职责**按代码仓库维度**划分，仅针对所申请的特定代码仓库生效。

## 2. 晋升标准和流程

### 2.1 晋升标准

#### 2.1.1 Committer晋升标准

优秀的贡献者可以**按仓库维度**申请成为Committer。申请时可参考以下条件：

- 活跃参与项目贡献3个月;
- 提出过多个有深刻见解的检视意见，体现对项目代码结构及演进方向具有足够理解;
- 在项目中贡献过大颗粒度特性，有助于项目在代码功能、质量、生态发展等方面进步；
- 项目中 Maintainer 或 Committer 提名，提名材料需附贡献清单（含 PR 链接、检视记录、核心贡献说明等）。

#### 2.1.2 Maintainer晋升标准

优秀的Committer通过SIG例会评审后，可晋升为Maintainer。方式分为两种：

- **新创建SIG的Maintainer**: 开发者通过寻找3个及以上志同道合的开发者，确定Maintainer候选人，通过技术指导委员会（TSC）申请新建SIG，经批准后成为新SIG的Maintainer。
- **已有SIG的Maintainer**: 已是对应SIG领域的Committer，可以由1名Maintainer提名，该SIG下全员Maintainer投票，获得半数及以上投票赞成且无反对票即可通过。

### 2.2 晋升投票流程

#### 2.2.1 Committer 晋升投票流程
- 提交提名：现任Maintainer/Committer以标题“[VOTE] New Committer xxx（仓库名称）”在对应提名仓库创建Issue，详细描述被提名人的社区贡献成果及晋升理由；
- 组内投票：对应仓库的所有Maintainer、现任Committer以“+1”（赞成）或“-1”（反对）形式投票，投票周期为7个自然日，逾期未投票视为弃权；
- 通过标准：需获得超半数投票人赞成，且无有效反对票（反对票需附具体理由，无理由反对视为无效）；
- 结果处理：若出现有效反对票，被提名人需在3个工作日内回应，双方协商达成一致后可延长投票周期3个自然日；协商无果则由TC介入仲裁；
- 公告授权：投票通过后，由Maintainer在社区发布公告（或SIG例会纪要），同步赋予对应仓库的Committer操作权限。

#### 2.2.2 Maintainer 晋升投票流程
- 提交提名：提名者以标题“[VOTE] New Maintainer xxx（SIG名称）”在对应SIG仓库创建Issue，附被提名人的贡献清单及提名理由；
- 组内投票：该SIG下所有现任Maintainer参与投票，投票周期为7个自然日，逾期未投票视为弃权；
- 通过标准：获得半数及以上投票赞成，且无有效反对票（反对票需附具体理由）；
- 结果处理：若有有效反对票，按Committer晋升流程的协商仲裁规则执行；
- 公告授权：投票通过后，在社区及TC同步晋升公告，赋予SIG维护管理权限，更新社区核心成员名单。


## 3. 非活跃成员退出标准和流程

### 3.1 非活跃成员定义

下表列出了不同角色或组织被认定为非活跃的判定标准。

| 类型 | 判定标准 |
| --- | --- |
| Committer | 连续6个月以上未参与代码审核、PR 提交或设计评审等活动。 |
| Maintainer | 连续3个月以上未参与社区管理活动，未召开、参与 SIG 会议，或连续6个月以上未在技术委员会(TC)交流 SIG 项目进展。 |

### 3.2 非活跃成员退出流程
#### 3.2.1 主动退出
- 成员自行撰写邮件，标题为“Inactive [角色] [姓名/GitCode ID]”（例：Inactive Committer xxx），发送至对应项目Maintainer邮箱，说明退出原因；
- Maintainer在SIG例会上对退出申请进行审视，形成退出结论；
- 结论确认后，由Maintainer移除其对应仓库/组织的操作权限，并在SIG内部公示。

#### 3.2.2 被动退出
- SIG Maintainer每季度收集非活跃成员名单，在SIG例会上审视确认非活跃对象，发起被动退出流程；
- 向非活跃成员发送提醒邮件，告知其非活跃状态及退出预警，给予15个自然日的申诉期；
- 申诉期内无回应或申诉未通过的，在SIG例会上确定退出结论；
- 结论确认后，在社区公示3个工作日无异议，移除其对应权限并更新台账。

### 3.3 复归机制
被动退出的成员，退出后3个月内重新满足对应角色的活跃标准，可由原提名者或本人向对应SIG申请恢复权限，经Maintainer SIG例会审核通过后，可恢复对应角色和权限。
