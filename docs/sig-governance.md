# Ascend社区特别兴趣小组（SIG-Special Interest Group）运作指导

本文档旨在为Ascend社区成员提供清晰、全面的 SIG（Special Interest Group）管理制度与操作指南。文档首先阐述 SIG 的核心运作框架，包括其组织架构、生命周期与决策机制。提供详细的操作指南，包括 SIG 的申请、日常运作、变更、终止及仓库管理等流程。

## 1. SIG 概述

SIG（Special Interest Group）是 Ascend社区中，由对特定技术领域充满热情的专家和贡献者组成的兴趣小组。每个 SIG 聚焦于一个明确的技术方向，设定清晰的工作目标，通过开放、透明的社区协作方式，推动 Ascend 社区在该领域的技术演进、项目开发与生态维护。

为鼓励更广泛的生态协作并明确社区运作边界，所有向 SIG 所辖仓库贡献代码的个人或实体，均须按照Ascend项目要求签署代码贡献协议。

## 2. SIG 运作说明

本章节定义了 Ascend 社区 SIG 的运作框架，是所有 SIG 必须遵守的核心准则，确保其健康、有序、高效地运作。

### 2.1 SIG 组织架构与角色职责

每个 SIG 均包含以下核心角色：

#### 2.1.1 Maintainer

SIG 的负责人，通常由若干名在该领域有深入见解和影响力的专家担任。

**职责**：

- 负责制定和推动 SIG 的技术愿景、目标和路线图。
- 召集和主持 SIG 例会，引导技术讨论和决策。
- 代表 SIG 与 TC/TSC 及其他 SIG 进行沟通协调。
- 定期与 TC/TSC 和社区交流 SIG 的工作进展、成果与风险。
- 维护 SIG 的健康运作，激励和发展社区成员。
- 负责 SIG 仓库权限等基础设置的管理。
- 负责社区Issue、例会等渠道开发者反馈问题、特性的分发处理。

#### 2.1.2 Committer

SIG 的核心贡献者，在特定代码仓库拥有写权限。

**职责**：

- 深度参与 SIG 的技术方案设计和讨论。
- 高质量地完成代码开发、提交和维护工作。
- 积极参与代码审核（Code Review），保障代码质量。
- 指导和帮助新成员融入 SIG。

#### 2.1.3 Contributor

对 SIG 所属领域感兴趣并参与贡献的社区成员。

**职责**：

- 参与 SIG 的公开会议和社区讨论。
- 通过提交 Issue，Pull Request 等方式为 SIG 做出贡献。

### 2.2 SIG 生命周期

SIG 的生命周期包括创建、运作、变更和终止四个阶段。

#### 2.2.1 创建 (Creation)

SIG 的创建需经过审批流程，以确保其技术方向符合 Ascend 社区的整体发展战略，且具备可行性和独特性。申请人需准备申请材料，通过所在项目TC（若无TC则向TSC申请）评审后方可成立。详细流程请参见[SIG管理流程](#3-sig-管理流程)部分。

#### 2.2.2 运作 (Operation)

SIG 批准成立后，即进入其生命周期中的核心活动阶段。在此阶段，SIG 遵循其运作指导，通过例会、代码贡献、技术讨论等方式积极开展工作，以实现其既定目标。SIG 的健康状况，如活跃度、贡献产出和社区影响力，将受到持续关注。

#### 2.2.3 变更 (Changes)
当 SIG 的工作范围、目标或Maintainer发生重大变化时，需要向所在项目TC（若无TC则向TSC申请）报备或申请审批。

- **Maintainer 变更**：需及时知会所在项目TC（若无TC则向TSC申请），并更新 SIG 组织信息和相关权限。
- **技术范围调整**：需与相关 SIG 充分沟通，并通过所在项目TC（若无TC则向TSC申请）评审，确保职责清晰，避免技术领域重叠。

#### 2.2.4 终止 (Termination)
当 SIG 完成其历史使命、长期不活跃或其技术方向不再符合社区发展需求时，可以被终止。

- Maintainer 可向所在项目TC（若无TC则向TSC申请）主动提交终止申请，并说明理由。
- 所在项目TC（若无TC则向TSC申请）也有权根据 SIG 的活跃度和贡献度，决定是否终止某个 SIG。
- 终止后，其SIG组织或归属的代码仓库将按流程进行退出处理。

### 2.3 决策机制与沟通

**决策机制**：SIG 内部决策遵循“共识优先”原则。对于重要技术决策，应通过公开会议充分讨论，力求达成共识。若无法达成共识，则由 Maintainer(s) 根据讨论情况做出决策，或所在项目TC（若无TC则向TSC申请）组织进行决策。

**沟通透明**：所有 SIG 的工作都应在公开渠道进行。

- **SIG例会**：定期召开公开会议，会议议程提前公布，会议纪要及时发布。(注：每个SIG独自制定例会计划，保证每月至少召开一次)
- **社区交流**：Maintainer需定期与所在项目TC（若无TC则向TSC）交流工作进展，对技术方向和工作内容进行审视。

## 3. SIG 管理流程

本章节提供了执行 SIG 运作规则的具体操作流程和方法。

### 3.1 申请新 SIG

#### 3.1.1 准备申请

在申请新 SIG 之前，申请人需要：

**阅读 SIG 管理制度**：深入理解本文档及[SIG 运作](#2-sig-运作)章节，明确 SIG 的运作规则、角色职责和生命周期。

**确认技术方向的唯一性和可行性**：

- 查阅已有的 SIG 列表，确保所申请的技术方向在社区中尚不存在。
- 确认所申请的技术项目能够最终转化为 Ascend 社区的新增组件或子项目。
- 如有疑问，可咨询所在项目TC（若无TC则向TSC）。

**准备 SIG 运作与目标**：参考[SIG 申请模板](./templates/sig/sig-template_cn.md) ，准备 SIG 介绍、定位、参与运作人员，并清晰地梳理 SIG 的工作目标、范围和预期成果。

#### 3.1.2 提交申请

- **创建 SIG 提案初稿**：按照 [SIG 申请模板](./templates/sig/sig-template_cn.md) 撰写提案。
- **提交 SIG 申请议题**：将申请议题提交给所在项目TC（若无TC则由TSC审视），发起 SIG 成立申请。

#### 3.1.3 所在项目TC（若无TC则向TSC） 审批提案

- SIG 发起人在所在项目TC（若无TC则向TSC申请）例行会议上介绍待新建的 SIG，阐述 SIG 的目标、意义和计划。
- 所在项目TC（若无TC则由TSC）对 SIG 的成立进行审批，评估其技术方向、可行性及与现有 SIG 的关系，并决定是否批准。
- 评审会议形成会议纪要，记录所在项目TC（若无TC则由TSC）的审批意见。该会议纪要是后续创建 SIG 的必要凭证。

#### 3.1.4 创建 SIG 组

所在项目TC（若无TC则由TSC）审批通过后，SIG 发起人可参考[Ascend项目社区SIG组权限管理方案](./sig-info-guidance.md)完成仓库配置。

### 3.2. 日常运作

#### 3.2.1 SIG 会议

SIG 应固定周期召开公开会议（线上或线下），保持信息同步和协作。会议日程及流程可查看[Ascend会议中心](https://meeting.ascend.osinfra.cn/)。会议召集人应提前预订会议，并通过会议日历、社区交流群组等方式通知会议举行的时间、地点和相关议程。会后召集人应及时组织、整理会议纪要，记录要点、讨论结果和决策事项并归档。

#### 3.2.2 SIG 管理与交流

- Maintainer 负责 SIG 的日常管理，包括工作内容审视、任务分配等。
- SIG 需定期向所在项目TC（若无TC则向TSC）和社区交流工作进展与成果，保持信息公开透明。
- 若 Maintainer 发生变动，需预先通知所在项目TC（若无TC则向TSC），并更新组织信息和仓库权限。

### 3.3 SIG 退出

如果SIG无法定期组织一定的人数或无法履行组织管理职责超过 6个月以上的，应启动退出程序。
